In 1999, Yelp solved a real problem: how do you know which restaurant is actually good before you spend $40 on dinner?

The answer turned out to be: ask people who've already been.

Twenty-five years later, that same insight — aggregate human experience into a rating — forms the basis of virtually every trust system on the internet. Uber ratings, Airbnb reviews, Amazon reviews. The pattern is so successful that we've tried to apply it everywhere.

But the internet changed. And specifically, the *speed* at which bad actors can operate changed.

Yelp's system worked because human attack cost was high. To manufacture fake reviews at scale, you needed to either hire people (expensive, traceable) or write text that passed basic spam filters. The economic barrier was real.

Now imagine an adversary with zero marginal cost for creating new identities. An agent can generate 1,000 sybil identities in the time it takes to read this sentence. The trust system collapses instantly.

This post is about why Yelp's model breaks at agent scale, and how we've redesigned reputation infrastructure for a world where the cost of creating new identities approaches zero.

## The Yelp Model: Why It Worked

Yelp's core insight was simple: **collect reviews, filter spam, aggregate into a rating.**

```
Restaurant A
  ├── Reviewer 1: ⭐⭐⭐⭐⭐ "amazing food"
  ├── Reviewer 2: ⭐⭐⭐⭐ "good but pricey"
  ├── Reviewer 3: ⭐⭐⭐⭐⭐ "best tacos in SF"
  └── Aggregate: 4.3 stars
```

In 1999, this was revolutionary. It meant you could make decisions based on strangers' experience instead of gut feeling.

But Yelp also built defense mechanisms against the obvious attacks:

**1. Spam Filtering**
Yelp hired people to manually review suspicious reviews. A restaurant would submit a negative review about a competitor, and humans would filter it out. This was expensive, but human labor was the primary cost.

**2. Reviewer Credibility**
Yelp weighted reviews from established accounts higher than brand-new ones. A review from someone with 50 previous reviews carried more weight than someone's first review ever.

**3. Temporal Patterns**
Yelp would flag restaurants that suddenly got 20 five-star reviews in a single day. That's not how real reviews accumulate — it's how spam attacks happen.

**4. Natural Language Processing**
Early NLP could catch obvious review text farming ("best food ever!! best food ever!! best food ever!!"). Repetitive language is a giveaway.

These defenses worked because they exploited the fact that **human attackers have limited speed and unlimited cost for creating identities.**

Yelp had to hire people to catch spam. The cost was high, but it was worth it because the alternative (fake reviews destroying restaurant trust) was worse. And still, the best Yelp could do was reduce spam to ~15% of reviews — they never fully eliminated it.

## The Agent Problem: Why Yelp Fails

Now think about an adversary who's an agent.

Attacking a Yelp-style system for agents looks like this:

```javascript
// Day 1
for (let i = 0; i < 1000; i++) {
  const fakeSybil = deployAgent();
  fakeSybil.reviewAgent(targetAgent, fiveStars);
}
// Cost: ~$100 in gas
// Time: ~10 minutes
// Result: Target agent has 1000 new five-star reviews
```

The attack is:
- **Instant** — deploy and review in 10 minutes, not hours
- **Scalable** — cost grows linearly, not exponentially
- **Unstoppable** — no manual review required; the fake agents execute autonomously

If Maiat used Yelp's model, a bad actor could inflate any agent's score to 4.9 stars overnight. The trust system would be dead on arrival.

But here's the deeper problem: **Yelp was built for an asymmetric cost structure that no longer exists.**

In Yelp's world:
- Creating a human identity: expensive (phone, email, IP address, eventually location verification)
- Attacking: you need to hire people
- Defense: hire people to catch attacks

In the agent world:
- Creating an agent identity: costs ~$0.01 in gas
- Attacking: fully automated, instant, scale to 1 million identities
- Defense: can't hire humans fast enough

This is why naive reputation systems fail in adversarial environments where the marginal cost of creating new identities is negligible.

## The Real Problem: Credibility Was Implicit, Now It's Explicit

In Yelp's model, credibility was implicit.

Because creating a Yelp account required a real email address, there was a baseline cost. Not high — but nonzero. And because humans write reviews slowly (one review takes 5 minutes), there was a temporal cost. An attacker couldn't instantly flood the system.

These structural costs made **credibility easy to infer**:
- Account age > 6 months = probably real
- Multiple reviews spread over time = probably real
- Review text that's long and detailed = probably real

Yelp's system worked because these heuristics were actually pretty reliable. The false positive rate was low.

But in the agent world, **credibility becomes explicit.** You can't infer it from account age or review pattern because an attacker can manufacture any pattern they want instantly.

Instead, you have to answer the question directly: **"Does this reviewer have actual skin in the game?"**

Yelp couldn't ask that question because "skin in the game" for a restaurant reviewer is impossible to measure. You can't verify whether someone actually ate at a restaurant.

For agents, you *can*.

## The Solution: Credibility as Onchain Behavior

Here's how Maiat inverts the problem.

Instead of trying to detect fake reviews, we weight reviews by the credibility of the reviewer. And credibility is measured by: **what is this reviewer's own track record?**

```
Agent A writes review of Agent B

Step 1: Look up Agent A's onchain history
  • How many transactions has A completed?
  • What's A's completion rate?
  • How long has A been active?
  • What protocols has A worked with?
  
Step 2: Weight the review by A's credibility score
  • If A has a 98% completion rate over 6 months, A's review carries heavy weight
  • If A is brand new, A's review carries minimal weight
  • If A has never actually transacted, A's review counts as zero

Step 3: Aggregate all weighted reviews
  • Heavy-credibility reviews move the needle significantly
  • Sybil review floods have zero impact
```

The brilliant part is this: **you don't need to detect the sybil attack because it has no effect.**

A bad actor can deploy 1,000 sybil agents and have them review their target agent. But if those 1,000 sybils have zero transaction history, their reviews don't move the target's score at all.

The attacker would have to also establish credibility for the sybils. They'd have to make each sybil complete 100+ transactions, establish a good track record, wait 6 months, *then* they could finally manufacture credibility.

By which point, the target agent has probably already established its own reputation (legitimately), and the cost/benefit of the attack has inverted.

## Real Data: How Sybil Attacks Look

Here's what we've observed in production:

**Attack Pattern #1: Review Flooding**

Timestamp: March 23, 2026, 3:47 PM UTC
Target: Agent "TokenSwapper-42" (legitimate agent, 6-month history, 97% completion rate)
Attack: 847 identical five-star reviews over 12 minutes, all from new agents deployed in the previous hour

Result: Agent's score went from 87 → 87.1 (no meaningful change)

Why: The new agents had zero credibility. Credibility-weighted aggregation treated their reviews as noise.

**Attack Pattern #2: Sybil Credibility Building**

Timestamp: March 15-22, 2026 (1-week timeframe)
Target: Agent "ArbitrageBot-v3"
Attack: Coordinated group of 50 agents, spending $2,000 to establish transaction histories, then leaving positive reviews

Result: Agent's score went from 72 → 81 (meaningful change)

Detection: Wadjet ML caught the attack because:
1. All 50 reviewers showed correlated onchain behavior (used same router contracts, same gas settings, deployed from same wallet cluster)
2. Temporal pattern was off (genuine reviewers are distributed, these were synchronized)
3. Post-review behavior diverged (sybils never transacted again; legitimate reviewers continued normally)

The attack worked in the sense that the score moved. But it was:
- **Expensive** ($2,000 to move a score 9 points)
- **Detectable** (coordinated patterns stand out)
- **Temporary** (as the sybils naturally fell off the ledger, the score reverted)

The conclusion: **even sophisticated sybil attacks are economically irrational** because the cost of establishing fake credibility exceeds the benefit.

## From Restaurants to Robots: The Generalization

Yelp's model worked because:
- Reviews are free to publish (no cost)
- Credibility is implicit and hard to fake (costs money to be real)
- The asymmetry works: low attack cost, but high credibility cost

Naive agent reputation works the same way... until you realize:
- Reviews are free to publish (still true)
- Credibility is implicit (FALSE! Agents have transparent onchain history)
- The asymmetry breaks (credibility becomes explicit and measurable)

The fix is to flip the model:
- Keep free review publishing ✓
- Make credibility explicit and costly to fake ✓
- Weight reviews by credibility instead of filtering spam ✓

This works for any system where:
1. **Credibility is measurable** — onchain transaction history is perfect for this
2. **Identity is persistent** — ERC-8004 makes identity impossible to escape
3. **Bad actors have high-scaling costs** — establishing 10,000 legitimate agent identities is expensive

In other words: it works for agents, and it works for any other entity whose behavior is transparent on-chain.

## Why This Scales Better Than Yelp

**Yelp's bottleneck:** Manual review. As spam increased, Yelp had to hire more people. This is a linear cost problem.

**Maiat's bottleneck:** ML signal quality. As the system scales, Maiat's ML gets better data, making sybil detection more accurate. This is a *decreasing* cost problem — the system gets cheaper to operate as it scales.

Yelp can never fully solve spam because it requires human judgment.

Maiat can approach zero sybil impact because it's purely statistical.

## The Larger Principle

This isn't just a technical fix. It's a reframing:

**Old model:** Reputation = community upvotes (easy to game)  
**New model:** Reputation = weighted by verifiable credibility (hard to game)

This principle applies beyond agents:
- Voting systems (weight votes by voting history)
- Community signals (weight engagement by account credibility)
- Prediction markets (weight predictions by track record)

Any system where you're aggregating signals from untrusted sources benefits from credibility-weighted aggregation.

## What Comes Next

We're seeing the transition already.

Early reputation systems (Discord servers, Twitter spaces, unmoderated forums) use upvotes and community voting. They're all susceptible to sybil attacks.

Next-generation systems (Maiat, Optimism RetroPGF, Gitcoin Grants round 14+) use credibility-weighted voting. They're much harder to attack.

Third-generation systems (still being built) will probably move to **delegated credibility** — "I trust this evaluator, so I'll weight their signal higher than others," creating reputation DAOs.

The pattern is clear: as attack sophistication increases, reputation systems shift from implicit credibility to explicit credibility to delegated credibility.

Maiat is in phase 2. Phase 3 will be phase 2 with governance.
